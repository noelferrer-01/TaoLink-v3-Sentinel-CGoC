## Transcript insights worth highlighting

- **"Sa recruitment, yun yung wala eh. Yun yung kailangan pumasok sa HRIS niyo na mga bagong empleyado"** — Recruitment is THE entry point. Explicitly stated multiple times by CG.
- **"Hindi ka naman pwede maglipat from A to B. Pag walang papel"** — "You can't transfer A to B without papers." Currently Operations sometimes does ad-hoc transfers without paperwork. This is the bug Sentinel must fix.
- **Calculation is simple (A + B); allocation is the hard part.** Payroll math is trivial; "which guard was at which client for how many days at what rate" is the real complexity.
- **Detachment admin presence varies** — some detachments have admin staff (handles clock-in entry); others don't. Clock-in workflow has to handle both.
- **CG has multiple existing systems** (partial HRIS, light inventory/firearms, executive system). None integrate with Sentinel per [0003](../../wiki/decisions/0003-relationship-to-existing-cg-system.md).
- **Operations role pivots** under the new model — logistics, inventory oversight, client communication, transfer requests (but not execution).
- **Hundreds of clients, each with own rate templates** — different break schedules, OT rules, night-shift differentials, Saturday premiums, emergency rates. Per-client billing tables, not per-guard. CONFIRMED again in transcript.

**Decisions taken without checking back:**

1. **Added Tailwind? No.** Stayed with a single hand-written `globals.css` + CSS variables. Tailwind setup is overkill for 11 screens and would dilute the editorial character. If the codebase later needs utility classes at scale, migration is mechanical.

2. **Per-cell DTR editor → Slice 2.** The plan's contract ("rows = active employees, columns = each day in the period. Each cell editable: status + (optional) timeIn/timeOut") is 150 cells × 4 fields each — non-trivial UI for Slice 1's demo path. Shipped a simpler "quick-fill per guard + per-period" interaction that makes the demo a 1-click happy path. Per-cell editing is logged as a Slice-2 todo.

3. **BIR 2316 → JSON download, not PDF.** PDF rendering depends on the IVB line-item schema fields (RDO/DOB/address) that don't exist yet — both logged as Slice-2 todos. JSON download is honest about what Slice 1 produces.

4. **Close-period action calls runPayroll inline** rather than relying on the existing `events.subscribe('dtr.period.closed', ...)` setup. Next.js server actions don't survive across requests in a way that keeps subscriptions warm; explicit inline call is reliable and matches the user's mental model ("close period → payslips appear").

5. **Forms collapse outside the toolbar's actions slot.** Initial Task-8.3 attempt put the open form *inside* `.page-toolbar-actions` and the card got squeezed into the right-aligned column. Restructured: the toolbar carries only meta on the left; the add form renders below the toolbar in full-width flow. Same pattern in clients, detachments, assignments.

6. **`.btn:disabled` styling** added during the Phase-8.8 gate after spotting that the "Assign a guard" button looked identical to its enabled self.

**UX bar adherence:** Plain-language labels everywhere ("Guards" not "Employees", "Time records (DTR)" not "DTR", "Take-home pay" + "Net pay" both surfaced on the payslip, "Government exports" not "Compliance exports"). Confirm-before-destructive on close-period (ochre button + bulleted consequences) and lock-pay-run. Empty states teach the next step (e.g. "/dtr → No guards to record for this period" links to /assignments). All forms surface single-line Zod errors in `.form-error` red.
