# SSS R-3 — Contribution Collection List — File Format Spec

**Source:** SSS Form R-3 (Contribution Collection List), Republic of the Philippines, Social Security System.
**Authoritative URL:** <https://www.sss.gov.ph/wp-content/uploads/2022/03/SSSForms_Contribution_Collection_List.pdf>
**Last verified:** 2026-05-24

---

## What this form is

The R-3 is the **quarterly remittance report** that every SSS-covered employer files alongside SS payments. It lists each employee with their monthly SS and EC (Employee Compensation) contributions for the three months of the quarter being reported.

A filed R-3 covers ONE quarter (Jan–Mar, Apr–Jun, Jul–Sep, or Oct–Dec) and is due **within the first 10 days of the month following the quarter**.

## Header fields (one set per filing)

| Field | Type | Notes |
| --- | --- | --- |
| Employer ID Number | string | 10-digit SSS employer number |
| Registered Employer Name | string | As registered with SSS |
| Telephone Number | string | |
| Registered Address | string | |
| Type of Employer | enum | `Regular` \| `Household` |
| Quarter Ending | date | Last day of the quarter (e.g. `2026-03-31`) |

## Per-employee row (one row per covered employee per quarter)

| Field | Type | Notes |
| --- | --- | --- |
| SS Number | string (10 digits) | Employee's SSS member ID |
| Surname | string | Family name first, comma, then suffix where present (e.g. `Dela Cruz`, `San Juan Jr.`) |
| Given Name | string | |
| Middle Initial | string (1 char) | |
| SS — 1st Month | decimal(12,2) | Combined ER + EE SSS contribution for month 1 of the quarter |
| SS — 2nd Month | decimal(12,2) | Month 2 of the quarter |
| SS — 3rd Month | decimal(12,2) | Month 3 of the quarter |
| EC — 1st Month | decimal(12,2) | Employee Compensation contribution (ER-only) for month 1 |
| EC — 2nd Month | decimal(12,2) | Month 2 |
| EC — 3rd Month | decimal(12,2) | Month 3 |
| Separation Date | date \| blank | `MM/DD/YYYY` if separated during the quarter; else blank |

**Termination row:** the final non-blank line is followed by a row that contains the literal text `Nothing Follows` per SSS reminder #2.

**Grand totals:** the last page of a paper R-3 carries a Grand Total + Payment Details block; for the electronic CSV that Sentinel produces, the totals row sums each of the six contribution columns.

## Sentinel CSV mapping (what `exportSSS_R3` produces)

Sentinel ships a CSV that mirrors the columns above. **Slice 1 scope:** the export is **per pay run**, not per quarter — the contributions for the pay run's calendar month are written into the corresponding month column (`1st Month` / `2nd Month` / `3rd Month`) and the other two contribution columns are left empty for that row. The HR clerk combines three months externally to produce the final R-3 filing.

A quarter-aggregating wrapper (`exportSSS_R3_Quarter(year, quarter)`) is deferred to a later slice.

### CSV column headers (exact, in order)

```
sss_number,surname,given_name,middle_initial,ss_1st_month,ss_2nd_month,ss_3rd_month,ec_1st_month,ec_2nd_month,ec_3rd_month,separation_date
```

### Fully-worked example row

Pay run for `2026-05-01 → 2026-05-15` (semi-monthly), employee Juan Dela Cruz, monthly basic salary ₱18,000, employee SS contribution for May = ₱675.00, employer SS = ₱1,350.00, EC = ₱30.00.

May is the 2nd month of the Apr–Jun quarter, so the export writes month-2 columns:

```csv
sss_number,surname,given_name,middle_initial,ss_1st_month,ss_2nd_month,ss_3rd_month,ec_1st_month,ec_2nd_month,ec_3rd_month,separation_date
0312345677,Dela Cruz,Juan,A,,2025.00,,, 30.00,,
```

(SS column = ER + EE combined = 675 + 1350 = 2025.00.)

After the last employee row, a totals row:

```csv
TOTAL,,,,,2025.00,,, 30.00,,
```

## Missing-data handling

The R-3 export does NOT block on missing SSS numbers. Rows where `hr_employees.sss_number IS NULL` are emitted with `sss_number=MISSING` and surfaced separately in the return value's `warnings` array so the HR clerk can fix-and-rerun before filing.

## Known constraints / followups

1. **Real-world filings use the SSS AMS (Automated Member System) Contribution Collection List submission**, which accepts either CSV import or DBF. The CSV column order Sentinel produces above is designed to be importable into AMS; an integration test against the AMS tool is a Slice 2 task.
2. **Adjustment R-3** (for amending a prior filing) requires checking the "Adjustment" box on the paper form. Sentinel's Slice 1 export does not flag adjustments; the clerk handles re-filing manually.
3. **The pre-printed R-3** (where SSS sends back the prior filing for verification) is not generated by Sentinel.

## Schema dependency

| Sentinel field | R-3 column |
| --- | --- |
| `hr_employees.sss_number` | SS Number |
| `hr_employees.last_name` | Surname |
| `hr_employees.first_name` | Given Name |
| `hr_employees.middle_name` | Middle Initial (first char) |
| `payslips.sss_ee` + derived ER share | SS — Nth Month |
| Derived EC share | EC — Nth Month |
| `hr_employees.terminated_on` (if within quarter) | Separation Date |

### Why ER and EC are derived, not stored

The Sentinel payroll engine only persists the **employee share** of each statutory contribution on `payslips` (`sss_ee`, `philhealth_ee`, `pagibig_ee`) — that's what affects net pay. The **employer share** of SSS, plus the **EC contribution** (employer-only), are part of company-side accounting, not the employee's payslip. We don't persist them per-payslip because:

- They're a deterministic function of the salary bracket on the date of the run.
- Storing them would double the contribution columns on every payslip just to satisfy filings.
- The compliance corpus (`gov_sss_contribution_table`) is the source of truth for the bracket → (EE, ER, EC) mapping; the exports module looks up ER and EC at export time using the same table the engine used.

This means R-3 export reproducibility depends on the SSS contribution table being **version-pinned by effective date** — already enforced in `modules/compliance` (each row carries `effective_from`).
